EXPLAINER · SAS-353 · BRIDGE PLAN-SUBMIT

Pourquoi on intercepte fetch('/submit')

La page plan.html contient déjà un bouton Submit qui fait fetch('/submit'). Tel quel, ce fetch part dans le vide. Le bridge sert à faire en sorte que ce même appel atterrisse dans la commande Tauri plan_submit, sans toucher au template.

TL;DR. L'iframe (origine asset://) et la WebView parente (origine tauri://) sont deux origines distinctes — la Same-Origin Policy empêche le parent de modifier directement le JS de l'iframe. Le seul canal autorisé entre les deux est postMessage. Le bridge implémente ce canal : la page de plan poste une demande au parent ; le parent appelle invoke('plan_submit', …) côté Rust ; la réponse repart par postMessage dans l'iframe.

1 · Le problème — pourquoi un fetch direct ne marche pas

La page de plan est servie depuis le système de fichiers via le protocole asset:// de Tauri. Elle ne tourne ni sur un serveur HTTP ni dans la même origine que l'app. Tout chemin réseau qu'elle tente est résolu par rapport à sa propre origine.

Naïf — sans bridge
L'iframe envoie fetch('/submit')
Le navigateur résout /submit contre l'origine de l'iframe : asset://localhost/submit. Aucun handler ne répond à cette URL → 404 / erreur réseau, le bouton Submit échoue silencieusement.
vs
Avec bridge
Le même fetch('/submit'), intercepté
Un petit script tourne dans l'iframe et remplace window.fetch. Quand l'URL est /submit, il envoie un postMessage au parent React, qui exécute plan_submit et renvoie une Response synthétique. La page ne voit pas la différence.
État du même appel fetch('/submit') selon que le bridge est installé ou non
⚠ SOP

On ne peut pas faire plus simple en écrivant directement dans iframe.contentDocument depuis React. L'iframe a une origine différente de la WebView parente — la Same-Origin Policy bloque l'accès cross-frame au DOM.

Concrètement : iframe.contentWindow.document.head.appendChild(script) lève une SecurityError. Le seul canal autorisé entre deux origines distinctes est postMessage.

2 · Les trois acteurs et leurs origines

Le bridge fait communiquer trois entités, chacune isolée dans son contexte. Les origines sont matérialisées par les couleurs et les bordures.

Architecture du bridge — qui parle à qui, et avec quel canal

Trois choses à retenir de ce schéma :

  • L'iframe et le parent sont deux mondes séparés. Même si la WebView est un seul process Chromium, l'iframe a sa propre origine (asset://localhost) — distincte de la WebView principale (tauri://localhost).
  • Le parent React ne peut pas appeler la page de l'iframe directement. Pas d'accès au document, pas d'eval, pas d'injection de <script> via JS. La flèche rouge en bas est la voie qu'on voudrait prendre mais que le navigateur refuse.
  • Seul postMessage traverse l'isolation. C'est l'API conçue exactement pour ça : passer des données sérialisables entre deux contextes d'origines différentes.

3 · Le flux complet, étape par étape

Que se passe-t-il vraiment quand l'utilisateur clique sur Submit dans le plan ? Six étapes, trois acteurs.

1
iframe · page de plan
L'utilisateur clique sur Submit
Le handler existant de plan.html appelle fetch('/submit', { method: 'POST', body: JSON.stringify(payload) }). C'est le code de la page, qu'on ne modifie pas.
2
iframe · script du bridge
L'intercepteur attrape l'appel
Le bridge a remplacé window.fetch. Il voit que l'URL est /submit, parse le body en JSON, génère un requestId unique, et poste au parent : parent.postMessage({ type: 'plan-submit-request', requestId, payload }, '*'). Il retourne une Promise en attente dans une Map.
3
webview parent · React
Le listener setupPlanSubmitBridge reçoit le message
L'écouteur window.addEventListener('message', …) filtre : il ignore tout ce qui ne vient pas de iframe.contentWindow (sécurité) et tout ce qui n'a pas la forme { type: 'plan-submit-request', requestId: string }.
4
process rust · tauri
Le parent appelle invoke('plan_submit', …)
L'IPC Tauri sérialise les arguments (repoPath, kind, slug, payload) et exécute la commande Rust. Celle-ci écrit docs/{kind}/{slug}/submission.json avec un JSON joli (indent 2 espaces), puis retourne le chemin absolu.
5
webview parent · React
Le parent renvoie le résultat à l'iframe
Le .then de l'invoke poste la réponse : iframe.contentWindow.postMessage({ type: 'plan-submit-response', requestId, ok: true, path }, '*'). Symétrique au cas ok: false, error si l'invoke rejette.
6
iframe · script du bridge
L'intercepteur résout la Promise de la page
Le bridge écoute aussi les message dans l'iframe. À réception, il récupère l'entrée pending via le requestId, et résout la Promise avec une Response synthétique : new Response(JSON.stringify({ ok: true, path }), { status: 200 }). La page voit un fetch réussi — son code de succès s'exécute normalement.
Séquence d'un Submit — du clic au fichier sur disque, et retour à la page

4 · Les deux chemins d'injection du script intercepteur

Le bridge a deux moitiés : côté parent (toujours actif, branché par useEffect) et côté iframe (le script qui réécrit window.fetch). C'est le script côté iframe qui a deux chemins de livraison possibles.

Chemin A · injection à chaud
tryInjectBridgeScript(iframe)
Sur le onLoad de l'iframe, le parent essaie d'écrire un <script> dans iframe.contentDocument. Échoue silencieusement en production (asset:// ≠ tauri:// = SOP), capturé par un try/catch. Utile uniquement si jamais les deux origines venaient à coïncider (tests, configuration alternative).
Chemin B · script embarqué (cible)
Le template plan-html inclut PLAN_SUBMIT_BRIDGE_SCRIPT
La constante exportée par plan-submit-bridge.ts est le code à coller tel quel dans un <script> de plan.html. Une fois embarqué dans le template, il s'exécute dans l'origine de l'iframe (autorisé) et installe l'intercepteur dès le chargement. Chemin canonique en production.
Deux façons de faire arriver le script dans l'iframe — l'injection à chaud est un best-effort, l'embarquement dans le template est le contrat réel

Pourquoi garder l'injection à chaud si elle ne marche pas en prod ? Parce que c'est gratuit (5 lignes derrière un try/catch) et que ça couvre les cas où le template n'a pas encore été mis à jour : si un jour on charge le HTML autrement (via srcdoc, depuis le même process Tauri, etc.) le bridge marche tout de suite, sans changer la page.

5 · Le contrat du protocole

Deux types de messages, formes strictes. Tout ce qui ne matche pas est ignoré. 

// Requête : iframe → parent
{
  type: 'plan-submit-request',
  requestId: string,    // unique, généré côté iframe
  payload: unknown      // JSON arbitraire à écrire dans submission.json
}

// Réponse succès : parent → iframe
{
  type: 'plan-submit-response',
  requestId: string,    // miroir de la requête
  ok: true,
  path: string          // chemin absolu écrit sur disque
}

// Réponse erreur : parent → iframe
{
  type: 'plan-submit-response',
  requestId: string,
  ok: false,
  error: string         // message d'erreur stringifié
}

Le parent filtre strictement sur trois choses avant d'invoquer :

  • event.source === iframe.contentWindow — refuse les messages venus d'ailleurs (autres iframes, fenêtre principale).
  • data.type === 'plan-submit-request' — refuse les autres conventions de messages.
  • typeof data.requestId === 'string' — refuse les payloads malformés.

Si l'un échoue, le message est ignoré silencieusement. Pas de surface d'attaque depuis une page tierce qui aurait été ouverte dans la même iframe.

À retenir

SOP
Deux origines distinctes (asset:// vs tauri://) = pas d'accès DOM cross-frame. postMessage est la seule API conçue pour ça.
Couplage zéro
La page plan.html continue à appeler fetch('/submit') comme si elle parlait à un serveur HTTP. Le bridge la trompe à son insu.
Sécurité
Le parent vérifie event.source et la forme du message avant d'invoke. Aucune surface d'attaque depuis une page tierce.
