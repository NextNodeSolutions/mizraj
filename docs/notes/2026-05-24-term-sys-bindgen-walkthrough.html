Pourquoi un crate -sys et pourquoi bindgen

En Rust, la convention pour interfacer une lib C est de scinder en deux crates :

1) Le crate -sys (celui-ci) contient uniquement les déclarations brutes — pour chaque fonction C il y a une signature Rust équivalente, et c'est tout. Pas de logique, pas d'abstraction. C'est ici, et seulement ici, que les blocs unsafe et extern "C" peuvent vivre.

2) Le crate haut-niveau (mizraj-term, viendra ensuite) wrappe les appels du -sys dans une API Rust sûre. Aucun unsafe n'est autorisé dans cette couche — l'unsafety est confinée par contrat.

Bindgen, c'est l'outil officiel rust-lang qui lit un header C (.h) et crache un fichier Rust avec toutes les extern "C" fn et les struct miroirs. Sans bindgen on devrait écrire à la main des centaines de signatures et les maintenir à jour à chaque release upstream. Avec bindgen, on régénère automatiquement à chaque cargo build à partir des headers vendorés.

L'avantage clé : impossible de désynchroniser la lib C et les bindings Rust. Si upstream change une signature, ça pète à la compil — pas à l'exécution dans un segfault opaque.

build.rs · 49 lignes — le cœur de la tâcheNEW

Un build.rs est un script Rust que Cargo exécute automatiquement avant de compiler la crate. Il s'organise en 5 étages :

① Trouver les headers (override-friendly)

Par défaut → vendor/include/. Override via env var LIBGHOSTTY_INCLUDE_DIR pour pointer vers une install système ou un checkout local. Pattern propre : sensible default + escape hatch documenté.

② Invalider le cache de build correctement

Ces println! ne sont pas du logging — c'est un protocole stdout que Cargo écoute. Ils disent : « rebuild si l'env var change, si build.rs change, ou si un header change ». Sans ça, modifier un .h ne déclencherait pas de régénération — bug subtil classique des FFI.

③ Validation défensive

Fail loud dès le premier header manquant, avec un message explicite qui dit comment override — au lieu d'un message bindgen cryptique 200 lignes plus loin.

④ Génération des bindings

Les .allowlist_* sont l'optimisation clé : ne génère que les symboles préfixés ghostty_ / GHOSTTY_. Sans ça, bindgen ressortirait tout ce que le préprocesseur a tiré (libc, types système, macros stdint…) → des milliers de symboles polluants à reviewer.

layout_tests(false) désactive les tests générés qui vérifient sizeof et l'alignement de chaque struct. Utile pour cross-compil et compile-time. Les headers C de libghostty-vt sont annoncés ABI-stables (cf. décision archi) — on n'a pas besoin de ces tests à ce stade.

⑤ Écriture dans OUT_DIR

OUT_DIR est un chemin que Cargo fournit à chaque crate. Le fichier généré n'est jamais commité — il vit dans target/ qui est ignoré par git. Garde le repo propre et garantit que bindings et headers sont toujours cohérents (puisque régénérés ensemble).

Ce qu'on n'a PAS vendoré : ghostty.h, le header de l'API d'embedding complète (qui demande de linker contre tout le runtime Ghostty). On ne veut que la surface standalone libghostty-vt — décision archi prise lors du spike précédent.

Vendoring vs git submodule : on choisit vendoring parce que c'est simple, hermétique, et que la CI n'a pas besoin de checkouts externes. Le revers c'est qu'il faut régénerer manuellement à chaque bump — mais le README.md fournit la commande curl exacte, et la fréquence sera trimestrielle au mieux.
