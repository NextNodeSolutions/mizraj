TermSink + CellFrame — un second sink qui transforme le flux PTY en grille rendue

« show me exactement tes changements comment ça fonctionne, pour que je valide l'architecture de ce que tu mets en place »

Ce que ça fait. Aujourd'hui le PTY émet ses octets bruts vers le frontend via TauriEventSink (event agent:output), et c'est <AgentLog> qui parse l'ANSI côté JS. Ces changements ajoutent une seconde voie en parallèle : un nouveau TermSink qui pousse les octets vers un thread dédié possédant un émulateur de terminal libghostty, et qui émet une grille de cellules déjà rendue (event agent:cells) — couleurs, attributs et glyphes résolus côté Rust.

Le point d'architecture à valider. Les deux sinks coexistent : on n'a rien retiré. agent:output (stream texte) et agent:cells (grille structurée) partent en même temps depuis le même fan-out. C'est volontaire, mais ça veut dire deux représentations du même flux qui transitent en parallèle — c'est ce choix-là qu'il faut acter.

L'architecture en un schéma

Le PTY reader ne connaît pas ses consommateurs : il itère sur un Vec<Arc<dyn OutputSink>> et appelle write() sur chacun. C'est le point de fan-out. Les deux sinks vivent derrière le même trait, sans se connaître.

existant nouveau synchrone (budget ~1 ms du sink) asynchrone (hors hot-path PTY)
Le PTY reader appelle write() sur chaque sink · TermSink ne fait que copier les octets sur un canal · tout le travail lourd (parse + snapshot + serde) vit sur le render thread

Pourquoi un thread dédié et pas du tokio::spawn ? Parce que Terminal et RenderState tiennent des pointeurs FFI bruts : ils sont !Send, donc impossible à faire traverser un runtime async multi-thread. Le thread possède le terminal et reçoit les octets par canal — personne d'autre n'y touche.

Le code, fichier par fichier

1 · sink.rs — le contrat (inchangé, mais c'est le pivot)

Rien n'a changé ici, mais c'est ce trait qui rend l'ajout possible sans toucher au PTY reader. Tout sink respecte Send + Sync et un budget de ~1 ms dans write() : le travail lourd doit partir sur un thread.

2 · commands.rs — le fan-out : deux sinks au lieu d'un

Le cœur du changement côté wiring. session_create construit maintenant un vec de deux sinks. Noter le app.clone() : le premier sink garde un clone du handle, le second prend l'original. C'est ici qu'on décide « les deux voies partent ensemble ».

@@ session_create @@ session_create_inner(&manager, &binary, cwd, move |id| { - vec![Arc::new(TauriEventSink::new(app, id.clone())) as Arc<dyn OutputSink>] + vec![ + Arc::new(TauriEventSink::new(app.clone(), id.clone())) as Arc<dyn OutputSink>, + Arc::new(TermSink::new(app, id.clone())) as Arc<dyn OutputSink>, + ] })
3 · term_sink.rs — le handoff non-bloquant

write() est sur le hot-path PTY : il ne fait que to_vec() + send() sur un canal mpsc. Le Mutex est là uniquement parce que mpsc::Sender est Send mais pas Sync, alors que le trait exige les deux. Le lock n'est tenu que le temps du passage d'octets.

4 · term_sink.rs — la boucle de rendu (coalescing + dirty)

Le thread possède le terminal. Il bloque sur le prochain chunk, puis draine tout ce qui est déjà arrivé : une rafale de 50 chunks → un seul frame. Le garde Dirty::Clean est l'intérêt même de RenderState : si rien n'a bougé (chunk de contrôle pur), on ne snapshot ni n'émet rien.

5 · cell_frame.rs — la forme du wire

Les types de agent-cockpit-term (Cell, Color) n'ont pas de derive serde — d'où ces types miroir sérialisables à la frontière Tauri. attrs voyage en bits bruts (1 octet) plutôt qu'en 6 booléens. CellFrame embarque le session_id dans le payload, exactement comme agent:output, pour qu'un seul event global serve toutes les sessions.

6 · pty.rs — la géométrie de départ partagée

Le child PTY et le terminal du render thread doivent démarrer sur la même géométrie, sinon la grille snapshottée ne correspond pas à ce que le programme croit afficher. On extrait deux constantes pub(crate) que les deux côtés importent.

@@ pty.rs @@ +pub(crate) const DEFAULT_ROWS: u16 = 24; +pub(crate) const DEFAULT_COLS: u16 = 80; @@ spawn @@ .openpty(PtySize { - rows: 24, - cols: 80, + rows: DEFAULT_ROWS, + cols: DEFAULT_COLS,

Mon seul changement de fond (cleanup /simplify)

Sur le diff existant, le /simplify a convergé sur une amélioration nette, appliquée : WireCell.ch passait de String à char.

@@ WireCell @@ pub struct WireCell { - pub ch: String, + pub ch: char, } @@ From<&Cell> @@ - ch: cell.ch.to_string(), + ch: cell.ch,

Pourquoi ça compte : cell.ch est déjà un char. Le to_string() allouait une String sur le tas par cellule — soit ~1 920 allocations par frame en 80×24, à chaque frame dirty. char sérialise en exactement la même string JSON ("H") : la forme du wire est identique, le frontend ne voit aucune différence. Pur gain d'allocations + de typage.

Deux points à acter (hors scope /simplify — pas corrigés)

Ce ne sont pas des nits de qualité mais des décisions d'archi/comportement. Je les remonte au lieu de les corriger en douce.

⚠️ La CI ne teste plus du tout tauri-app

Ajouter agent-cockpit-term en dépendance fait que le binaire de test de tauri-app doit linker libghostty — absent en CI. La parade a été --exclude tauri-app sur cargo test. Conséquence : tous les tests unitaires de tauri-app disparaissent de la CI — les anciens (tauri_sink, sink, path…) et les nouveaux (cell_frame, term_sink). Le job verify-libghostty ne fait que vérifier des checksums de headers, il ne lance aucun test.

FIX PROFOND (hors diff) : cible de test lib-only, ou un runner CI provisionné avec libghostty. À décider — c'est une régression de couverture réelle.

⚠️ Le Terminal du render thread n'est jamais redimensionné

session_resize redimensionne le master PTY uniquement. Le Terminal du thread reste figé à DEFAULT_ROWS/COLS. Les constantes règlent l'accord initial ; mais au premier resize, la grille rendue (agent:cells) désynchronise de la taille réelle du terminal.

DOMAINE : correctness/feature (pas /simplify). C'est la suite logique de la feature — propager le resize jusqu'au render thread (canal d'événements, ou méthode resize sur OutputSink).

✓ Ce qui est propre par ailleurs

  • Reuse : pas de duplication — les types wire sont nécessaires (agent-cockpit-term n'est pas serde-aware), DEFAULT_ROWS/COLS sont partagés via pty.rs.
  • Mutex<Sender> : justifié et documenté (Sender est !Sync) — pas un défaut.
  • Hot-path : aucun travail lourd dans write() ; tout est sur le render thread, derrière le canal.
  • Le build a passé typeck + borrowck (l'échec local n'est que le link de libghostty, absent ici comme en CI).
