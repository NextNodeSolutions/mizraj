Explainer · Rust · Tauri command

list_plans — scan des artefacts docs/ d'un projet

Commande Tauri synchrone qui découvre les fichiers plan.html et *.html dans docs/interviews/ et docs/plans/ d'un projet cible, et retourne une liste de métadonnées prête à alimenter le menu Plans de la sidebar.

Une seule entrée : repo_path: String — chemin absolu vers la racine d'un projet.

Une seule sortie : Vec<PlanEntry> avec { kind, slug, path, title, mtime } par artefact trouvé. kind = "interview" ou "plan". Erreurs filesystem propagées en Err(String).

Scope : P4-01 fait le scan. Le champ title vaut le slug pour l'instant — l'extraction réelle est SAS-347 (P4-03).

Signature et contrat de retour

Côté Rust — src-tauri/src/commands/list_plans.rs 
#[derive(Debug, Serialize)]
#[serde(rename_all = "camelCase")]
pub struct PlanEntry {
    pub kind:  String,  // "interview" | "plan"
    pub slug:  String,
    pub path:  String,  // absolu, depuis repo_path
    pub title: String,  // = slug (placeholder P4-03)
    pub mtime: u64,     // secondes Unix epoch
}

#[tauri::command]
pub fn list_plans(
    repo_path: String,
) -> Result<Vec<PlanEntry>, String>

Ce qui ne fait pas partie de P4-01

Titre humain : le champ title contient le slug. L'extraction réelle (<title> ou <h1> du plan.html, fallback slug) est livrée par SAS-347 / P4-03 — c'est sa raison d'être.

Tri : la sortie n'est pas triée. L'ordre vient de read_dir (donc filesystem-dépendant). Le tri par mtime desc sera géré côté front (P4-02 — menu Plans sidebar).

Watcher : la commande est pull uniquement. La détection live d'un nouveau fichier sera P4-09 (docs/interviews/) et P4-10 (docs/plans/).

Pas d'IPC du côté JS pour l'instant : la commande est enregistrée dans le invoke_handler mais aucun composant React ne l'appelle encore. C'est P4-02 qui consomme.
