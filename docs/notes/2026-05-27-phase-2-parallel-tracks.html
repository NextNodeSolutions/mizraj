Ce qu'il reste à faire pour avoir claude dans une fenêtre de l'app

Oublie les phases, les tracks et les D-machin. Il y a 5 chantiers. Quatre te bloquent pour avoir un terminal qui marche. Le cinquième est à côté. Voilà.

Aujourd'hui
Le binding libghostty est fini
La crate Rust agent-cockpit-term sait déjà parser les bytes ANSI en grille de cells. C'est la pièce la plus risquée. Elle est mergée.
Objectif
Claude tourne dans une fenêtre de l'app
Tu cliques « nouvelle session », tu vois claude démarrer, tu tapes dedans, l'output s'affiche. Pas de scrollback persistant, pas de multi-pane. Juste la boucle qui marche.

Les 5 chantiers qui restent

à faire
CHANTIER 1

Lancer un process dans un PTY

Spawner claude dans un pseudo-terminal, pouvoir lire son output et lui envoyer des keystrokes. Crate portable-pty, la même que wezterm/alacritty.

Rust · tauri-app effort I4
à faire
CHANTIER 2

Gérer les sessions côté backend

Un objet qui garde la liste des PTYs vivants, qui démarre/arrête les process proprement, et qui meurt bien quand l'app quitte (SIGTERM puis SIGKILL après 2s).

Rust · tauri-app effort I6
à faire
CHANTIER 3

Faire transiter les bytes entre Rust et React

Les channels Tauri v2 : un canal par session, qui transporte les cells dans un sens, les keystrokes dans l'autre. C'est l'autoroute qui relie le backend au frontend.

Rust + React effort I4
à faire
CHANTIER 4

Afficher la grille dans un canvas

Le composant React <TerminalPane> qui reçoit les cells et les dessine dans un <canvas>. Plus la capture des keystrokes qui repartent vers le PTY.

React · canvas 2D effort I6
en cours
CHANTIER 5 · à côté

Les wrappers Git (libgit2)

Les fonctions diff_session, diff_working_tree, worktree_list… dans la crate agent-cockpit-vcs. Indépendant du terminal — tu peux le faire en // ou plus tard, ça ne bloque rien.

Rust · agent-cockpit-vcs effort I4

Le 4 a besoin du 3, qui a besoin du 2, qui a besoin du 1. C'est une chaîne. Tu ne peux pas paralléliser les chantiers 1→2→3→4 entre eux.

Ce que tu peux faire EN MÊME TEMPS

✓ Vraiment en parallèle

  • Chantier 1 (PTY) en backend la chaîne principale
  • Chantier 5 (libgit2 wrappers) à côté, zéro contention
  • Squelette du Chantier 4 avec des cells mockées

3 trucs en // si tu as deux têtes (toi + un agent Claude par exemple). Sinon prends 1 + 5 en alternance.

✗ Faut attendre

  • Chantier 3 (channels) attend le Chantier 2
  • Chantier 4 final (wire-up) attend le Chantier 3
  • Scrollback persistant SQLite après le terminal vivant
  • Multi-pane, raccourcis Cmd+K polish, après le palier

TL;DR

Tu fais le Chantier 1 (PTY). En même temps un agent Claude peut bosser sur le Chantier 5 (libgit2). Quand 1 est fini, tu enchaînes 2 → 3 → 4. À la fin du 4, claude tourne dans une fenêtre de ton app. Multi-pane, scrollback, raccourcis : tout ça vient après.
